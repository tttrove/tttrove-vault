# 极客技术博客风格知识笔记生成 Skill

本 Skill 基于对 geek-blogs.com 多篇文章的深度分析，提炼其文笔风格、文章结构和知识库格式，
用于复用生成任意技术知识点的高质量中文笔记或手册。

---

## 一、博客风格画像（必须内化）

### 1.1 叙述视角

- **主角是工具/技术本身**，以第三人称客观介绍；但使用"你"直接面向读者，不用"我们"。
- 口吻：严谨但不生硬，专业但不晦涩，像一位有经验的工程师在手把手讲解。
- 避免自夸和情感渲染，陈述事实、场景和操作即可。

### 1.2 标题格式

```
主题关键词 + 冒号 + 副标题定位词

示例：
  Linux 安装与使用 Screen：终端会话管理神器全指南
  Git Rebase 深度解析：从原理到最佳实践
  Docker 网络模式全指南：从基础到进阶配置
```

副标题定位词候选（根据内容深度选用）：
- 完全指南 / 全指南 / 全解析
- 从基础到进阶 / 从入门到精通
- 深入理解 XX 及其应用
- XX 原理与最佳实践

### 1.3 引言三段式结构

每篇文章的引言必须包含三个层次，**顺序固定**：

```
① 痛点/场景描述（1-2句）：点出读者面临的真实问题或使用场景
② 工具/技术介绍（1-2句）：定义主角，说明它能解决什么
③ 核心优势 bullet list（3-5条）：每条格式为 **特性名**：一句话说明
④ 文章预告（1句）：本文将从 XX、XX、XX 等方面……帮助读者……
```

**示例引言（Screen 文章）：**
> 在 Linux 系统管理或开发过程中，我们经常需要在终端中执行长时间运行的任务……
> 但传统终端有一个痛点：**一旦终端关闭或 SSH 连接断开，正在运行的任务会被终止**。
>
> **GNU Screen** 是一款终端复用工具，它能解决上述问题……
>
> Screen 的核心优势：
> - **持久性**：进程在后台持续运行，不受终端断开影响。
> - **多任务**：一个终端窗口管理多个独立会话。
> - **可恢复性**：意外断开后，重新连接即可恢复工作状态。
>
> 本文将从安装、基础用法、高级功能、最佳实践及故障排除等方面……

  * [5.2 最佳实践](#)

## 三、格式规范（逐条遵守）

### 3.1 Markdown 格式约定

| 元素 | 规范 |
|------|------|
| **加粗** | 关键术语首次出现、核心概念、注意事项 |
| `行内代码` | 命令名、参数、文件路径、变量名 |
| 代码块 | 所有可执行命令必须用代码块，注释写在 `#` 后 |
| 编号章节 | 一级 `## 1.`，二级 `### 1.1`，三级 `#### 1.1.1` |
| 目录 | 文章开头必须有完整目录，用嵌套列表 |
| 引用块 | 安全提示/重要警告用 `> **提示**：` 格式 |
| 表格 | 命令对照、参数说明、状态枚举适合用表格 |

### 3.2 代码块规范

```bash
# 每个代码块都要有注释说明用途
sudo apt update          # 更新软件包索引
sudo apt install -y xxx  # 安装工具（-y 自动确认）
```

- 多发行版安装命令：按 Debian / RHEL / Arch / Fedora / openSUSE 分子节
- 示例输出必须标注含义（用注释 `# 说明` 或文字紧随其后解释）

### 3.3 括号注释风格

技术名词后用括号补充说明，格式统一：
- 英文全称：`终端复用（Terminal Multiplexer）`
- 中文解释：`FHS（Filesystem Hierarchy Standard，文件系统层次结构标准）`
- 简写提示：`GNU Screen（简称 Screen）`

### 3.4 列表风格

核心优势 / 最佳实践列表格式：
```
- **特性名**：一句话说明，可跟括号补充场景。
```

场景列表用 `#### 场景 N：标题` 标注，内容包含：
1. 1-2 句描述适用场景
2. 代码示例（带注释）

## 四、写作语言规范

### 4.1 常用句式模板

**开篇引入：**
- `在 [场景] 中，我们经常需要 [动作]……但 [痛点]。`
- `[工具名] 是一款 [类型] 工具，它 [核心功能]。`
- `与 [对比对象] 相比，[工具名] 的 [特点] 更为 [形容词]。`

**功能介绍：**
- `[功能] 允许用户 [动作]，并支持 [子功能]。`
- `通过 [参数/命令]，可以 [效果]。`
- `[工具] 的核心 [概念] 是 [定义]，其工作原理如下：`

**最佳实践：**
- `始终用 [方式] 执行 [操作]（如 [举例]），避免 [反例]。`
- `建议从 [基础] 开始实践，逐步探索 [进阶]。`

**总结结尾：**
- `[工具名] 是一款 [轻量/强大/灵活] 的 [类型]，通过 [核心机制] 解决了 [问题]。`
- `掌握 [工具名] 后，你可以：`
  - `[能力1]`
  - `[能力2]`

### 4.2 禁止使用的表达

❌ 避免主观情感词：「非常棒」「简直太好了」「令人惊叹」  
❌ 避免第一人称：「我认为」「我们建议」→ 改用「建议」「推荐」  
❌ 避免模糊表述：「很多情况下」→ 改用「在 SSH 远程管理场景中」  
❌ 避免口语化：「其实」「说白了」「其实挺简单的」

## 五、执行流程

收到用户请求后，按以下步骤生成笔记：

### Step 1：解析主题

从用户输入提取：
- **主角**：工具名 / 概念名（如 `Git Rebase`、`TCP/IP 协议栈`）
- **深度**：入门 / 进阶 / 完全指南（默认"完全指南"）
- **受众**：开发者 / 运维 / 初学者（影响措辞难度）
- **侧重**：偏操作手册 / 偏原理理解 / 偏踩坑排错

### Step 2：规划章节

根据主题类型选择章节配置：

| 主题类型 | 必含章节 | 可选章节 |
|---------|---------|---------|
| CLI 工具 | 引言、安装、基础用法、高级功能、最佳实践、故障排除、总结 | 配置文件详解、与竞品对比 |
| 编程概念 | 引言、核心原理、使用方式、常见模式、反模式、总结 | 性能考量、版本差异 |
| 系统知识 | 引言、工作原理、结构组成、配置实践、故障排除、总结 | 历史背景、安全加固 |
| 框架/库 | 引言、安装配置、核心概念、API 速查、最佳实践、总结 | 与同类框架对比 |

### Step 3：生成内容

按模板逐章输出，每章节遵守：
1. 章节开头 1 句定位句（说明本章覆盖范围）
2. 核心内容（代码块 + 注释 + 说明文字交替）
3. 关键点用 bullet list 或表格归纳

### Step 4：收尾检查

输出完成后，内部自检：
- [ ] 标题格式是否符合「主题：副标题」格式
- [ ] 引言是否包含痛点→工具→优势→预告四要素
- [ ] 目录是否完整且与章节一致
- [ ] 代码块是否全部带注释
- [ ] 最佳实践是否编号且每条有标题
- [ ] 故障排除是否有「原因」和「解决」两段
- [ ] 总结是否有 bullet list 形式的"你可以"

## 六、快速参考：各章节写法示例

### 安装章节写法
```markdown
## 2. 安装 XXX

### 2.1 检查是否已安装

可通过以下命令确认：

    xxx --version

若输出版本号（如 `xxx 1.2.3`），则已安装。若提示 `command not found`，则需手动安装。

### 2.2 在主流环境中安装

根据你的系统，使用对应包管理器：

#### Debian / Ubuntu

    sudo apt update && sudo apt install -y xxx

#### RHEL / CentOS 8+

    sudo dnf install -y xxx

#### Arch Linux

    sudo pacman -S xxx
```

### 故障排除章节写法
```markdown
## 6. 故障排除

### 6.1 [错误描述，如：权限不足导致无法启动]

错误提示：

    Permission denied: /var/run/xxx.sock

原因：当前用户未加入 `xxx` 用户组，无访问权限。  
解决：执行以下命令后重新登录：

    sudo usermod -aG xxx $USER
    newgrp xxx  # 立即生效（无需重启）
```

### 总结章节写法
```markdown
## 7. 总结

[工具名] 是一款 [轻量/强大] 的 [类型]，通过 [核心机制] 有效解决了 [问题场景]。
其 [核心特性1] 和 [核心特性2] 使其成为 [目标用户] 的必备工具。

掌握 [工具名] 后，你可以：

- 在 [场景1] 中 [能力1]，不再担心 [痛点1]。
- 用 [方式2] 高效 [能力2]。
- 跨 [维度] 无缝 [能力3]。

建议从 [基础功能] 开始实践，逐步探索 [进阶功能]，结合最佳实践持续提升效率。
```

## 七、输出说明

- **默认输出格式**：Markdown（`.md`），适合 Obsidian、Notion、GitHub 等知识库
- **输出长度**：完全指南约 3000-6000 字，速查手册约 800-1500 字
- 若用户指定"速查手册"或"cheat sheet"，精简为：引言 + 命令速查表 + 常见场景 + 注意事项
- 若用户指定"深度原理篇"，在核心概念章节增加原理图（用 ASCII 或 Mermaid 代替）和内核/底层解释
